UX/UI 디자인(3)
UX 리서치

사전 질문(15분)
서비스를 사용하면서 "이건 사용자를 전혀 이해하지 못하고 만들었구나"라고 느낀 경험이 있나요? 있다면 그 이유는 무엇이었나요?

"사용자 입장에서 생각해봐"라는 말을 들었을 때, 정말로 사용자 입장이 될 수 있다고 생각하나요?

사전 질문 토론
1. 사용자 중심을 고려하지 않은 서비스 경험
다양한 사례(공공기관 앱, 관공서 앱, C커머스, 스마트워치 등)에서 불편함과 비직관적 UI, 반복 인증, 용어 혼란 등을 경험.
서비스 흐름 단절, 숨겨진 기능, 낚시성 이벤트, 비효율적 메뉴 구조 등 사용자 중심 설계 부족으로 불만을 느낌.
기업 입장에서 복잡성을 유지하거나 편의보다 내부 요구를 우선시한 결과로 지적.
2. '사용자 입장에서 생각해봐'에 대한 인식
일부는 페르소나, 사용자 여정, 관찰을 통해 근접 가능성을 봄.
다수는 완전한 사용자 입장 대체는 불가능하며, 진정한 사용자가 되어야 본질적 이해 가능하다고 판단.
제작자가 자신이 만든 서비스의 한계를 벗어나기 어려운 점도 언급.
사용자 경험 개선을 위해 관찰, 인터뷰, 직접 체험 등이 중요하다고 강조.

Case: 테스트하고 바뀌게 된 것
디자인 스프린트를 통해 제품의 초기 버전 디자인을 완료
내부적으로는 만족스러운 결과 - 유의미한 성과가 있을 것을 기대
개발 전에 그래도 사용성 테스트를 해보자는 의견에 추가로 사용성 테스트를 진행
다른 팀의 내부 직원 대상 -> 주변 지인들 대상 -> 완전 상관없는 타겟 모집
실제 사용성 테스트 결과
우리가 기대했던 것 보다 제품이 무엇인지 이해를 하지 못한다
우리가 기대했던 것 보다 제품 사용에 불편함을 느낀다
우리가 기대했던 것 보다 무엇을 얻을 수 있는지 발견하지 못한다

만든 사람은 사용자의 마음을 근본적으로 이해할 수 없다 - UX 리서치의 핵심 필요성

UX 리서치의 본질 - 추측이 아닌 발견
왜 우리의 확신은 자주 빗나가는가?
"이 버튼은 여기 있는 게 당연하지"
"사용자도 이 플로우를 자연스럽게 따라갈 거야"
"이 정도 설명이면 충분해"
But 우리는 이미 너무 많이 알고 있습니다.

지식의 저주
PM의 머릿속:
모든 기능의 위치를 암기
기술적 제약사항 이해
비즈니스 로직 파악
개발 히스토리 인지
사용자의 머릿속:
이게 뭘 하는 앱이지?
어디를 눌러야 하지?
이렇게 하는 게 맞나?
왜 안 되지?

고객 분석과 UX 리서치의 명확한 구분
상황: 온라인 쇼핑몰의 장바구니 이탈률 70%
고객 분석이 밝혀내는 것:
"배송비가 비싸서 구매를 포기한다"
"더 싼 곳이 있나 비교하러 간다"
"지금 당장 필요하지 않아서 미룬다"
→ What & Why에 대한 답
UX 리서치가 찾아내는 것:
"배송비가 마지막 단계에서야 나타난다"
"총 금액이 어디에 표시되는지 모르겠다"
"다음 단계로 가는 버튼을 못 찾겠다"
→ How에 대한 답
💡 핵심: 고객 분석은 '무엇이 문제인가?'를 찾고, UX 리서치는 '어떻게 해결할 것인가?'를 검증합니다.

리서치 마인드셋: 겸손한 탐구자 되기
Beginner's Mind (초심자의 마음)
잘못된 자세:
"내가 사용자보다 제품을 더 잘 안다"
"이렇게 하면 당연히 이해할 것이다"
"복잡하다고 느끼는 건 익숙하지 않아서다"
올바른 자세:
"나는 사용자에 대해 모른다"
"내 가정이 틀릴 수 있다"
"사용자가 전문가다"

해결: 다양한 사용자 그룹 포함
해석 편향: 원하는 대로 해석하는 것
해결: 다른 사람의 해석도 들어보기

1. 맥락적 탐구 - 진짜 환경에서의 진짜 행동
사용자가 일하고 생활하는 실제 환경에서 그들을 관찰하고 대화하는 리서치 방법
일반적인 테스트 환경:
조용한 사무실
WiFi 완벽
양손 자유롭게 사용
시간 여유 충분
실제 사용 환경:
비 오는 밤 길거리
한 손엔 우산
배터리 10%
급하게 택시 필요
환경이 달라지면 사용자의 행동도, 니즈도 완전히 달라집니다.

수행 방법 - 준비 단계
1. 관찰 목표 설정
Bad: "사용자가 앱을 어떻게 쓰는지 보자"
Good: "첫 구매 과정에서 어디서 막히는지 찾자"
2. 참가자 선정
초보 사용자: 온보딩 문제 발견
헤비 유저: 고급 기능 니즈
이탈 사용자: 근본 문제 파악
3. 도구 준비
관찰 노트 템플릿
녹화 장비 (동의 필수)
보상 (기프티콘 등)

수행 방법
2. 실행 단계
시작하기: "평소처럼 사용해주세요. 저는 배우러 왔습니다."
첫 20분은 그냥 지켜보기
사용자가 편안해질 시간 필요
자연스러운 행동 관찰
관찰 중:
침묵 유지 (개입 최소화)
중요한 순간 메모
표정과 행동 관찰
함께 해석하기
막힐 때: "지금 무엇을 찾고 계신가요?"
우회할 때: "평소에도 이렇게 하시나요?"
포기할 때: "무엇이 어려우셨나요?"

수행 방법
3. 분석 단계
패턴 찾기
여러 사용자의 공통된 행동
반복되는 문제 상황
인사이트 도출
"왜 이런 행동을 할까?"
"진짜 니즈는 무엇일까?"

맥락적 탐구의 핵심 원칙
1. Master-Apprentice Model
잘못된 접근:
연구자: "왜 그 버튼을 눌렀어요?"
사용자: "어... 그냥요?" (방어적)
올바른 접근:
연구자: "방금 하신 것을 다시 보여주시면서 설명해주실 수 있나요?"
사용자: "아, 제가 보통 이렇게 하거든요..." (교육적)

맥락적 탐구의 핵심 원칙
Focus: 무엇을 볼 것인가?
특정 태스크 (예: 상품 검색 → 구매)
특정 상황 (예: 출근길 사용)
Context: 환경과 상황 기록
물리적 환경
시간적 제약
감정 상태
Partnership: 함께 탐구하기
판단하지 않기
호기심 갖기
배우는 자세
Interpretation: 의미 찾기
"왜 그렇게 했을까?"
"무엇이 불편했을까?"
"어떻게 개선할까?"

2. 일기 연구(Diary Study) - 시간의 흐름 속 진실
사용자가 일정 기간 동안 자신의 경험을 기록하도록 하는 리서치 방법입니다.
피트니스 앱을 예 - 일반적인 테스트가 보여주는 것:
"운동 기록 기능 편해요!"
"UI가 깔끔해요!"
Diary Study가 보여주는 것:
Day 1-3: "신나서 매일 기록!"
Day 7: "오늘은 좀 귀찮네..."
Day 14: "알림 끄는 법이 뭐지?"
Day 21: "앱 삭제"
진짜 이야기는 시간이 지나야 나타납니다.

진행 절차
1. 연구 설계: 무엇을 알고 싶은가? 
모호한 질문: "사용자들이 우리 앱을 어떻게 쓰는가?"
구체적인 질문: "신규 사용자가 우리 앱의 핵심 기능을 발견하고 일상에 통합하는 과정은 어떻게 되는가?"
2. 기간 설정: 얼마나 길게?
1주일: 첫인상과 초기 적응
2-4주: 습관 형성 과정
1-3개월: 장기 사용 패턴

진행 절차
3. 기록 방법: 어떻게 기록할까?
구조화된 일일 질문:
오늘의 앱 사용 경험 (매일 저녁 9시)
1. 오늘 앱을 몇 번 열었나요?
2. 가장 유용했던 순간은?
3. 불편했던 점이 있다면?
4. 내일도 사용할 것 같나요? (1-5점)
자유 형식:
음성 메모
스크린샷 + 코멘트
짧은 일기

좋은 프롬프트의 조건
구체적이고 행동 중심적
❌ Bad Prompts:
"오늘 앱을 사용했나요?"
"만족스러웠나요?"
"개선점이 있나요?"
✅ Good Prompts:
"앱을 열게 된 순간의 상황을 설명해주세요"
"막혔던 부분이 있다면 스크린샷과 함께 설명해주세요"
"다른 방법으로 해결한 것이 있다면 무엇인가요?"

단계별 프롬프트 예시
Week 1: 탐색기
"오늘 발견한 새로운 기능이 있나요?"
"이해가 안 되는 부분이 있었나요?"
Week 2: 적응기
"자주 사용하는 기능은 무엇인가요?"
"더 빠른 방법을 찾았나요?"
Week 3-4: 습관기
"없으면 불편한 기능은?"
"친구에게 추천한다면 뭐라고 설명하시겠어요?"

실제 사례로 이해하기
사례: 언어 학습 앱의 30일
연구 목표: 왜 사용자들이 중도 포기하는가?
참가자: 영어 공부를 시작한 직장인 10명
Week 1 (희망 단계)
"하루 10분이면 충분하다니 좋네요!"
"레벨 테스트 재밌어요"
Week 2 (현실 직면)
"어제 못했는데 오늘 20분 해야 한다고?"
"회사에서는 소리 내기 어려워요"

실제 사례로 이해하기
Week 3 (위기)
"연속 학습 끊겨서 의욕 상실"
"난이도가 갑자기 어려워졌어요"
Week 4 (분기점)
포기 그룹: "시간이 없어서..."
지속 그룹: "친구랑 경쟁하니까 재밌네요"
인사이트:
2주차가 가장 위험한 시기
사회적 요소가 지속성의 핵심
유연한 학습 일정 필요
개선안:
2주차 특별 지원 프로그램
학습 친구 매칭 기능
"쉬는 날" 허용 옵션

성공적인 Diary Study를 위한 팁
1. 참가자 동기부여
문제: 시간이 지날수록 대충 쓰게 됨
주간 체크인: "이번 주 기록 잘 봤습니다!"
단계별 보상: "벌써 절반이나 오셨네요!"
흥미로운 질문 순환: 매주 다른 포커스
2. 기록 부담 줄이기
너무 많이 요구하면: 참가자 지침, 형식적인 답변, 중도 포기
적절한 균형: 하루 5분 이내, 핵심 질문 3-4개, 선택적 추가 질문
3. 실시간 인사이트 활용
매주 중간 분석을 통해
흥미로운 패턴 발견
추가 질문 기회
참가자와 대화 시작점

3. 사용성 테스트 - 디자인 검증의 핵심
실제 사용자가 여러분의 제품으로 특정 작업을 수행하는 과정을 관찰하고 분석하는 것
개발자/디자이너의 저주 - '초보자의 눈'을 잃어버림

실전 가이드: 8단계 프로세스
Step 1: 도메인 전문가 식별
해당 분야 5년 이상 경력자를 LinkedIn, 컨퍼런스, 논문 등에서 검색합니다.
실무 경험과 소통 능력을 갖춘 전문가를 2-3명 선정합니다.
Step 2: 인터뷰 일정 조율
전문가의 스케줄에 맞춰 1시간 정도의 인터뷰 시간을 요청합니다.
인터뷰 목적과 다룰 주제를 명확히 전달하여 동의를 구합니다.
Step 3: 인터뷰 스크립트 작성
도메인 기초 지식을 학습하고 전문가 수준에 맞는 질문을 준비합니다.
산업 현황, 주요 과제, 미래 전망 순으로 논리적 흐름을 구성합니다.
Step 4: 사전에 질문 전달
핵심 질문 3-5개를 미리 공유하여 전문가가 준비할 수 있게 합니다.
구체적인 사례나 데이터를 준비해올 수 있도록 안내합니다.

실전 가이드: 8단계 프로세스
Step 5: 인터뷰 진행
녹음 동의를 구하고 편안한 분위기에서 대화를 시작합니다.
전문 용어는 즉시 확인하고, 구체적 사례를 요청하며 깊이 있게 탐구합니다.
Step 6: 후속 인터뷰 진행
첫 인터뷰 분석 후 생긴 추가 질문으로 2차 인터뷰를 진행합니다.
다른 관점이나 검증이 필요한 부분을 확인하고 전문가 네트워크를 확장합니다.
Step 7: 정기적 체크인 수행
프로젝트 진행 상황을 공유하며 방향성을 검증받습니다.
새롭게 발생한 이슈나 변경사항에 대한 전문가 의견을 구합니다.
Step 8: 결과 평가 및 종합
여러 전문가의 의견을 교차 분석하여 편향을 줄입니다.
도출된 인사이트를 팀과 공유하고 실행 가능한 액션 아이템을 정리합니다.

실제 사례
Case: 의료 예약 시스템 개선 - 병원 예약 앱을 만드는 스타트업
인터뷰 대상:
병원 원무과장 (15년 경력)
의료 IT 컨설턴트
보험사 심사 담당자
핵심 발견:
원무과장 인터뷰에서: "환자들은 '빈 시간에 예약'하고 싶어하지만, 의사들은 '특정 질환을 모아서' 보고 싶어해요. 이 갈등이 스케줄링의 핵심입니다."
의료 IT 컨설턴트: "EMR 시스템과 연동 없이는 의미가 없어요. 하지만 병원마다 EMR이 다 달라서..."
보험사 담당자: "실손보험 청구 프로세스를 이해하지 못하면 환자 경험의 절반을 놓치는 거예요."
결과:
단순 예약이 아닌 통합 플랫폼으로 피벗
EMR 연동을 MVP에서 제외하고 단계적 접근
보험 청구 도우미 기능 추가

2. 과거 경험에 갇힐 수 있다
해결: "과거엔 그랬지만 지금은?" 질문

3. 지나친 복잡성
해결: "일반 사용자에게 설명한다면?"

7. 경쟁 분석 - 남의 실수에서 배우기
시장에서 검증된 패턴을 발견하고
사용자가 이미 익숙한 것을 이해하며
우리만의 차별점을 찾는 과정
1. 바퀴를 재발명하지 않기
"결제 플로우를 처음부터 설계해야 할까?" → 이미 수백만 명이 사용하는 검증된 패턴이 있습니다
2. 사용자의 기대치 이해
사용자: "다른 앱에서는 스와이프로 삭제하던데..." → 업계 표준을 무시하면 학습 비용 증가
3. 포지셔닝 전략
"우리는 어떤 점에서 다른가?"
"어떤 gap을 채울 수 있는가?"

⚠️ 주의할 함정
❌ 틀린 접근
"A사 기능 베끼기"
"B사보다 기능 더 많이"
"C사 디자인 그대로"
✅ 올바른 접근
"왜 A사는 이렇게 했을까?"
"사용자는 이 패턴에 얼마나 익숙할까?"
"우리 상황에 맞는 인사이트는?"
🤔 핵심 질문
"왜 사용자가 우리 대신 경쟁사를 선택할까?"

Step-by-Step 가이드
Step 1: 경쟁사 리스트 만들기
직접 경쟁사 (Direct Competitors)
같은 문제를 같은 방식으로 해결
같은 타겟 사용자
예: Spotify vs Apple Music
간접 경쟁사 (Indirect Competitors)
같은 문제를 다른 방식으로 해결
다른 타겟이지만 겹치는 니즈
예: Netflix vs YouTube
대체재 (Substitutes)
전혀 다른 방식의 해결책
예: 운동 앱 vs 헬스장

경쟁 분석의 한계와 주의점
1. 맹목적 추종
"대기업이 하니까 맞겠지"
→ 그들의 context와 우리 context는 다름
2. 기능 경쟁
"A사는 50개 기능, 우리는 60개!"
→ 사용자는 핵심 가치를 원함
3. 표면적 분석
"UI가 예쁘네, 따라하자"
→ 왜 그런 선택을 했는지가 중요
