UX/UI 디자인(4)
정보 구조와 탐색 설계

사전 질문(15분)
여러분이 자주 사용하는 서비스에서 메뉴나 기능의 위치가 '왜 여기 있지?'라고 의아했던 경험이 있나요? 그렇게 배치된 이유는 무엇일까요?

사용자가 원하는 기능이나 정보를 쉽게 찾도록 하려면, 무엇을 기준으로 메뉴와 콘텐츠를 구성해야 할까요?

사전 질문 토론
Q1. '왜 여기에 있지?' 싶은 메뉴나 기능 위치에 대한 경험과 그 이유
서비스를 쓰다 보면 내가 자주 쓰는 기능이 너무 깊숙하거나 이상한 위치에 있을 때가 많다.
기업 입장에서 사용자가 불편을 느끼더라도 전략적으로 배치한 것 같다는 생각이 든다.
예를 들어 해지 버튼을 찾기 어렵게 만들거나, 자주 쓰는 기능 대신 자사 서비스 노출을 우선하는 경우도 있다.
결국 사용자를 위한 설계라기보단 기업의 목적을 위한 설계처럼 느껴질 때가 많다.
Q2. 사용자가 원하는 정보를 쉽게 찾도록 메뉴/콘텐츠를 구성하는 기준
나는 메뉴나 콘텐츠를 구성할 때 가장 먼저 사용자의 목적과 흐름을 떠올린다.
자주 쓰는 기능은 눈에 잘 띄는 곳에 있어야 하고, 예상 가능한 위치에 있어야 한다고 생각한다.
복잡한 구조보다 직관적인 구조가 더 중요하고, 익숙한 용어와 정돈된 구성은 기본이다.
결국 기능 중심이 아니라 사용자 입장에서 어떻게 쓰게 될지를 먼저 상상해봐야 한다.

Case: 기능을 아무리 열심히 만들어도…
셀프케어 앱 서비스의 PM으로 처음 왔을 때 확인한 것 - 19개의 기능이나 존재한다는 것
하지만 주요 기능을 제외한 나머지 기능은 아무도 사용하고 있지를 않음
이를 어떻게 해결해야 할까?

모든 기능에 대해 나열을 한 뒤 카드 소팅을 진행
우리의 생각과 다르게 계층 구조와 카테고리가 다르게 묶여있음을 확인
심지어 계층 레벨이 달라 어떤 상위 기능은 다른 메뉴의 하위 기능에서 사용할 수 있음
기능 사용의 혼란함은 대부분 정보 인식의 잘못에 오는 경우

정보 구조(Information Architecture)의 본질
"도서관에서 책을 찾는 것처럼, 디지털 공간에서 정보를 찾을 수 있게 하는 기술"
정보 구조란 무엇인가?
콘텐츠를 조직화하고 레이블링하여 사용자가 쉽게 찾고 이해할 수 있게 하는 것
보이지 않지만 모든 디지털 경험의 뼈대를 이루는 구조
왜 중요한가?
아무리 좋은 콘텐츠도 찾을 수 없으면 존재하지 않는 것과 같음
잘못된 IA는 비즈니스 손실로 직결
Amazon: 로딩 시간 0.1초 증가 → 매출 1% 감소
“하지만 더 큰 문제는 아예 찾지 못하는 것”

IA가 매출에 미치는 영향
가상 Case: 대형 이커머스
카테고리가 100개 이상
평균 클릭 수 5.2회
장바구니 포기율 68%
IA 개선 작업:
Card Sorting으로 재분류
메인 카테고리 12개 → 7개
인기 상품 shortcuts 추가
검색 자동완성 강화
결과:
평균 클릭 수: 5.2 → 3.1회
전환율: 2.3% → 3.8%
월 매출 23% 증가

사용자는 어떻게 정보를 찾는가?
Information Foraging Theory: 정보 사냥꾼으로서의 사용자
핵심 비유: 사용자는 야생에서 먹이를 찾는 동물과 같다
최소한의 에너지로 최대한의 가치를 얻으려 함
"정보 냄새(Information Scent)"를 따라감
정보 냄새(Information Scent)란?
정의: 링크나 네비게이션 요소가 목적지의 콘텐츠를 얼마나 잘 암시하는가
사용자가 보는 것 (링크 텍스트, 아이콘)
사용자의 사전 지식과 경험
현재의 목표와 맥락

정보 냄새(Information Scent)
강한 정보 냄새의 예:
"겨울 패딩" → 따뜻한 겨울 외투를 찾을 수 있을 것 같음
장바구니 아이콘 → 내가 담은 상품을 볼 수 있을 것 같음
약한 정보 냄새의 예:
"Products" → 무슨 제품인지 알 수 없음
"Learn More" → 무엇에 대해 더 알게 되는지 불명확

Information Scent 강화 전략
1. 명확한 레이블링
구체적이고 설명적인 언어 사용: ❌ "Services" ✅ "웹사이트 제작 서비스"
사용자의 언어 사용: ❌ "Knowledge Base", ✅ "도움말 센터"
2. 맥락 정보 제공
트리거 워드 활용: 사용자가 찾는 키워드를 포함
미리보기 제공: 호버 시 간단한 설명 표시
비주얼 힌트: 아이콘, 이미지로 내용 암시
3. 정보 냄새의 일관성
클릭 전 기대와 클릭 후 경험이 일치해야 함
"미끼 제목" 피하기 - 신뢰 상실로 이어짐
💡 3초 규칙: 사용자는 3초 안에 원하는 링크를 찾지 못하면 이탈 확률이 급증

"사용자는 3번 이상 클릭하는 것을 싫어한다"는 통념
Steve Krug의 연구 결과:
"사용자는 많은 클릭을 신경 쓰지 않는다.
각 클릭이 쉽고, 명확하며, 올바른 방향으로 가고 있다는 확신을 준다면."
클릭 수보다 중요한 것: Interaction Cost
총 인터랙션 비용 = 목표 달성까지 필요한 모든 노력
물리적 노력 (클릭, 스크롤, 타이핑)
인지적 노력 (읽기, 이해하기, 결정하기)
감정적 비용 (불안, 짜증, 혼란)
실제 사례:
1번 클릭 but 50개 메뉴 중 선택 → 인지 부담 ⬆️
5번 클릭 but 매번 2개 중 명확한 선택 → 인지 부담 ⬇️

사용자의 정보 탐색 패턴
1. Known Item (알고 있는 정보)
"내 주문 내역 어디서 보지?"
빠른 접근 경로 제공이 핵심
2. Exploratory (탐색적 검색)
"선물하기 좋은 향수가 뭐가 있을까?"
브라우징 지원과 영감 제공 필요
3. Re-finding (다시 찾기)
"어제 봤던 그 상품..."
최근 본 상품, 검색 기록 활용
4. Exhaustive (철저한 조사)
"5만원 이하 무선 이어폰 전체"
강력한 필터와 비교 기능 필수

어떻게 사용자는 온라인 정보를 읽는가
"사용자는 읽지 않는다. 스캔한다."
읽기 vs 스캔하기
사용자는 페이지의 22%만 읽음 (평균)
목적: 원하는 정보를 최대한 빨리 찾기
방식: 비선형적, 선택적 주목

F-패턴: 가장 보편적인 스캔 패턴
F자 모양의 시선 이동:
상단 수평 이동 (첫 번째 가로선)
조금 아래서 짧은 수평 이동 (두 번째 가로선)
왼쪽 수직 스캔 (세로선)
설계 시사점:
가장 중요한 정보는 상단과 왼쪽에
첫 2-3단어가 핵심
부제목과 불릿 포인트 활용

스캔 친화적 디자인 전략
1. 시각적 우선순위 구축:
크기: 중요할수록 크게
색상: 대비로 주목도 높이기
위치: 상단 좌측이 황금 지대
공백: 여백으로 그룹핑
움직임: 제한적으로 사용

스캔 친화적 디자인 전략
2. 스캔 가능한 텍스트
짧은 문단: 한 문단 = 하나의 아이디어
의미 있는 부제목: 내용을 요약하는 제목
불릿 포인트: 리스트로 정보 구조화
강조 표시: 볼드체로 키워드 하이라이트
Front-loading: 중요한 정보를 앞에
3. 역피라미드 구조
가장 중요한 정보 (결론)
뒷받침하는 세부사항
배경 정보나 맥락

F-패턴의 진화와 변형
Layer Cake 패턴 (레이어 케이크)
제목-이미지가 반복되는 구조
예: 블로그, 뉴스 사이트
각 섹션의 제목만 스캔
Spotted 패턴 (점박이)
특정 요소만 선택적으로 주목
예: 가격 비교 사이트
숫자, 키워드만 찾아봄
Commitment 패턴 (몰입)
관심 있는 콘텐츠는 꼼꼼히 읽음
예: 상품 상세 설명
초반 스캔 → 관심 → 정독

IA의 4대 구성 요소
1. Organization System (조직 체계)
콘텐츠를 어떻게 분류하고 구조화할 것인가
우리가 Card Sorting으로 발견하는 것
2. Labeling System (레이블링 체계)
정보를 어떻게 표현할 것인가
사용자가 이해하는 언어로
3. Navigation System (네비게이션 체계)
정보 사이를 어떻게 이동할 것인가
길을 잃지 않도록
4. Search System (검색 체계)
정보를 어떻게 찾을 것인가
네비게이션의 안전망

Organization System: 정보를 조직하는 방법
1. 정확한 조직화 (Exact)
알파벳순: 이름을 알 때 (예: 연락처)
시간순: 최신/오래된 순 (예: 뉴스)
지리적: 위치 기반 (예: 매장 찾기)
2. 모호한 조직화 (Ambiguous) - 더 어렵지만 더 유용
주제별: 내용으로 분류 (예: 뉴스 섹션)
작업별: 하고 싶은 일로 분류 (예: "송금하기")
대상별: 사용자 유형별 (예: 개인/기업)
은유적: 현실 세계 모방 (예: 바탕화면, 폴더)

Organization System: 정보를 조직하는 방법
"정확한" 조직화를 쓸 때:
사용자가 찾는 것을 정확히 알 때
객관적 기준이 필요할 때
"모호한" 조직화를 쓸 때:
브라우징하며 탐색할 때
사용자가 뭘 원하는지 모를 때
💡 Best Practice: 보통은 둘을 조합해서 사용

조직화 체계: 어떻게 분류할 것인가
Database 방식 = 엑셀 표처럼
하나의 상품을 여러 기준으로 정리
예: 나이키 운동화
브랜드별 → 나이키
가격대별 → 5-10만원
용도별 → 런닝화
장점: 원하는 방식으로 찾기 쉬움
Hypertext 방식 = 위키피디아처럼
관련 있는 것끼리 링크로 연결
예: 운동화 페이지 → 런닝 팁 → 마라톤 대회
장점: 탐험하듯 새로운 발견
실무 선택:
쇼핑몰? Database 방식
블로그/매거진? Hypertext 방식

사용자가 만드는 분류 - Folksonomy
전문가 분류 vs 사용자 태깅
like 인스타그램
공식 카테고리 없음
사용자가 #해시태그로 분류
결과: 예상 못한 연결과 트렌드 (#OOTD, #먹스타그램)
실무 적용법:
기본 틀은 제공: 큰 카테고리는 PM이
세부는 자유롭게: 태그로 사용자가
인기 태그 활용: 자동 추천, 트렌드 반영
주의할 점:
#사진 #포토 #photo → 같은 의미 다른 태그
해결: 자동 통합 또는 추천

Labeling System: 이름이 반이다
레이블링이 중요한 이유
Case: "내 정보" vs "마이페이지" vs "프로필"
같은 기능, 다른 이름
사용자 혼란 → 찾지 못함 → 이탈
1. 텍스트 레이블
제목: 페이지나 섹션의 내용을 요약
링크: 목적지를 명확히 설명
인덱스 용어: 검색을 위한 키워드
2. 아이콘 레이블
장점: 언어 장벽 없음, 공간 절약
단점: 해석이 다양할 수 있음
해결책: 텍스트와 함께 사용

좋은 레이블 만들기
일관성이 핵심:
같은 개념 = 같은 용어
전체 서비스에서 통일된 어조
사용자 언어 사용:
내부 용어 ❌
업계 전문용어 ❌ (타겟이 전문가가 아니라면)
사용자 인터뷰에서 나온 표현 ✅

Navigation System의 구조 - 네비게이션의 3+1 유형
1. Global Navigation
사이트 전체에서 접근 가능
주요 섹션으로의 관문
일관된 위치 (보통 상단/좌측)
2. Local Navigation
현재 섹션 내에서 이동
맥락적 옵션 제공
사이드바, 탭 형태
3. Contextual Navigation
콘텐츠 내 관련 링크
"See Also", "관련 상품"
자연스러운 탐색 유도

Navigation System의 구조 - 네비게이션의 3+1 유형
+1. Supplemental Navigation
사이트맵, 인덱스, 가이드
다른 방법이 실패했을 때 안전망
모바일에서의 변화 - 공간 제약으로 인한 진화
햄버거 메뉴 (논란은 있지만 표준화)
하단 탭 바 (엄지 친화적)
제스처 네비게이션

네비게이션 설계의 디테일
로컬 네비게이션 잘 만들기 - "메인에서는 길 찾기 쉬운데, 안으로 들어가면 미로예요"
해결책 3종 세트:
현재 위치 표시: 메뉴에 색깔이나 밑줄
빵부스러기: 홈 > 남성복 > 상의 > 티셔츠
로컬 메뉴: 현재 섹션의 다른 페이지들
연관 콘텐츠의 힘
넷플릭스가 잘하는 것:
"이 영화를 본 사람들이 또 본 영화"
체류 시간 ↑, 이탈률 ↓
어떻게 구현?
간단: 같은 태그 가진 콘텐츠
고급: 사용자 행동 데이터 분석

Search System: 검색은 네비게이션의 보완재
"검색을 쓴다 = 네비게이션이 실패했다"
하지만 없으면 안 되는 안전망
특히 콘텐츠가 많을수록 필수
1. 검색 인터페이스: 검색창 위치 (표준: 우상단), 자동완성, 검색어 제안
2. 검색 알고리즘: 어떤 순서로 보여줄 것인가, 오타 교정, 동의어 처리
3. 검색 결과 표시: 관련도 vs 최신순 vs 인기순, 필터와 정렬 옵션, 검색어 하이라이트
4. No Results 처리: "결과가 없습니다" ❌, "이런 건 어떠세요?" ✅, 대안 제시가 핵심

검색 똑똑하게 만들기
1. 자동완성의 마법
사용자: "갤ㄹ..."
시스템: "갤럭시 S24 찾으시나요?"
2. 오타는 내가 고쳐줄게
입력: "아이펀 15"
이해: "아이폰 15"

검색 똑똑하게 만들기
3. 검색 결과 없을 때
❌ "결과가 없습니다" (끝)
✅ "아이폰 15 레드는 없지만..."
   다른 색상 아이폰 15
   비슷한 가격대 스마트폰
   인기 스마트폰
측정해야 할 것:
검색 → 클릭 비율
재검색 비율 (첫 검색 실패)
검색 포기율

Card Sorting: 사용자에게 분류를 맡기자
📌 카드 소팅이란?
사용자가 제공된 개념들을 그룹으로 묶고 이름을 붙이는 과정을 통해 정보 구조(IA)를 사용자 관점에서 설계하는 UX 리서치 방법
🎯 언제 사용하나?
메뉴 구조나 콘텐츠 분류 체계를 만들 때
정보 탐색의 효율성을 높이고자 할 때
사용자 관점의 구조를 확인하거나 개선할 때
🧠 기대 효과
사용자 친화적인 IA 설계
탐색성 향상 → 이탈률 감소
조직 내 이해관계자 설득 자료로 활용 가능

Card Sorting 진행 절차
1단계: 준비 (1-2일)
✅ 카드 준비
30-60개의 주요 콘텐츠/기능 선정
명확하고 중립적인 이름으로 작성
전문용어 배제, 사용자 언어 사용
✅ 참가자 모집
타겟 사용자 15-20명
다양한 사용 패턴 대표
✅ 도구 선택
오프라인: 인덱스 카드, 포스트잇
온라인: OptimalSort, UserZoom, Maze

Card Sorting 진행 절차
2단계: 실행 (참가자당 30-60분)
Open Card Sorting
"이 카드들을 비슷한 것끼리 그룹으로 묶어주세요.

"이 카드들을 준비된 카테고리에 넣어주세요.
어디에 넣을지 헷갈리는 것은 표시해주세요."

3단계: 분석 (1-2일)
카드 간 그룹핑 빈도 계산

70% 이상 함께 묶임 → 같은 카테고리로
30% 이하 → 분리 필요
4단계: 적용
새로운 IA 구조 도출
애매한 항목은 다중 경로 제공
Tree Testing으로 검증 진행

Card Sorting의 실제
음식 배달 앱을 만든다고 했을 때: 주문하기, 주문내역, 찜한 가게, 리뷰 쓰기, 쿠폰함, 포인트, 배달 추적, 가게 검색,1:1 문의, 주소 관리, 결제수단 관리, 알림 설정
일반적인 PM의 분류:
주문 관련: 주문하기, 주문내역, 배달추적
계정 관리: 주소 관리, 결제수단, 알림 설정
고객 지원: 1:1 문의, 리뷰
혜택: 쿠폰함, 포인트
실제 사용자의 분류:
지금 시킬 때: 주문하기, 가게 검색, 주소 관리
시킨 후에: 배달추적, 주문내역
내 것들: 찜한 가게, 쿠폰함, 포인트, 결제수단
불만있을 때: 1:1 문의, 리뷰 쓰기

Tree Testing: 만든 구조가 정말 찾기 쉬운가?
📌 트리 테스팅이란?
메뉴나 정보 구조의 탐색 흐름을 텍스트 기반 트리(Tree)로 구성하여, 사용자가 특정 정보를 얼마나 잘 찾는지 테스트하는 방식
🔍 무엇을 검증하나?
사용자가 예상한 위치에 정보가 있는지
분류 체계가 명확하고 직관적인지
메뉴 깊이, 항목 이름, 구조 등이 탐색에 어떤 영향을 미치는지
📈 활용 예시
리뉴얼 전 IA 구조의 문제 진단
카드 소팅 후 설계된 IA의 검증
여러 IA 버전 중 비교 실험 (A/B 트리 테스팅)

Tree Testing 진행 절차
1단계: 준비 (1일)
✅ 트리 구조 준비
Card Sorting 결과로 만든 IA 구조
텍스트만으로 계층 구조 작성
시각적 요소 모두 제거
✅ 태스크 작성 (5-8개)
좋은 예: "겨울 외투 무료 반품 정책을 확인하세요"
나쁜 예: "반품 정책을 찾으세요" (너무 직접적)
✅ 참가자 모집
30-50명 (정량적 분석 가능)
Card Sorting과 다른 참가자

Tree Testing 진행 절차
2단계: 실행 (참가자당 15-20분)
태스크 제시
텍스트 트리에서 클릭하여 탐색
정답이라고 생각하는 위치 선택
다음 태스크 진행

Tree Testing 진행 절차
3단계: 분석 (1일)
핵심 지표
Success Rate: 정답 찾은 비율 (목표: 70% 이상)
Directness: 첫 클릭 정답률 (목표: 60% 이상)
Time: 평균 소요 시간 (목표: 20초 이내)
문제 유형별 대응
첫 클릭 실패 → 레이블 변경
깊은 탐색 후 포기 → 구조 단순화
여러 경로로 분산 → 중복 배치 고려
4단계: 개선
문제 발견된 레이블 수정
구조 조정 후 재테스트
최종 IA 확정

Tree Testing의 실제
"여성 겨울 코트 50% 할인 상품을 찾아주세요"

결과:
40%는 여성의류 > 아우터로 감
60%는 SALE > 시즌오프로 감
정답은? 둘 다에 있어야 함!
온라인 쇼핑몰 메뉴 구조:
홈
├── 여성의류
│   ├── 상의
│   ├── 하의
│   └── 아우터
├── 남성의류
├── 가방/잡화
└── SALE
    ├── 타임세일
    └── 시즌오프

Tree Testing 실무 가이드
❌ 이렇게 하지 마세요:
"상품 정보를 찾으세요" → 너무 모호함
"여성의류 > 아우터에서 코트를 찾으세요" → 답을 알려줌
✅ 이렇게 하세요:
"딸아이 겨울 외투를 사려고 합니다. 어디서 찾을 수 있을까요?"
실제 상황을 구체적으로 설명
메뉴에 있는 단어를 그대로 쓰지 않음

"그 페이지 아직도 있어요?"
"이거 작년에 없앤 거 아니었나?"
"이 내용 3군데에 다 있네요?"
Content Inventory = 집 안 물건 목록 만들기

Visual Hierarchy: 중요한 걸 중요하게 보이게
스퀸트 테스트 (Squint Test)
아무 웹사이트나 열기
눈을 찌푸려서 흐릿하게 보기
뭐가 보이나요?
잘 된 사이트:
로고, 핵심 메시지, CTA 버튼이 보임
중요한 순서대로 보임
못 된 사이트:
뭐가 뭔지 모르겠음
다 똑같이 흐릿함

Visual Hierarchy 만드는 도구
1. SCALE (크기)
계층적 크기 조절은 가장 중요한 정보에 가장 큰 크기를 부여
덜 중요한 정보는 상대적으로 축소되어 시각적 순서를 명확히 함
“큰 것부터 본다”는 인간의 인지 특성을 활용한 전략.
2. CONTRAST (대비)
밝고 어두운 색, 단순한 도형과 복잡한 형태 간의 차이는 시각적 우선순위를 만듦
디자인 내에 대비가 클수록 먼저 인식되고 더 오래 기억
HEADLINE은 복잡한 형태로 대비되어 눈에 띔

Visual Hierarchy 만드는 도구
3. DIRECTION (방향)
일반적인 정렬 방식(가로, 수평)에서 벗어난 세로 또는 대각선 정렬은
사용자 시선을 끌고 정보 흐름을 명확히 만듦
방향성을 주면 사용자의 탐색 흐름이 설계된 대로 유도
4. POSITION (위치)
페이지 내 위치만으로도 자동적인 위계가 생성
일반적으로 좌→우, 상→하로 읽는 환경에서
좌상단이 가장 먼저 인식되고, 우하단이 가장 늦게 인식됨
중요한 정보는 페이지의 상단 왼쪽에 두는 것이 효과적

Footer Design의 기초: 페이지 하단의 기회
Footer = 웹페이지의 바닥글 영역
모든 페이지 하단에 공통으로 들어가는 영역
보통 회색 배경으로 구분
회사 정보, 링크 모음, 법적 고지 등 포함
왜 Footer가 중요한가?
일반적인 생각: "어차피 아무도 안 봐"
실제 데이터: 방문자의 25%가 Footer까지 스크롤
더 중요한 사실: Footer를 보는 사용자는 "적극적 사용자"

Footer Design의 심리학
Footer 도달 사용자의 3가지 유형
1. Mission-Driven (목적형)
특정 정보를 찾아서 도달
주로 찾는 것: 회사 정보, 채용, Contact
설계 포인트: 명확한 카테고리와 레이블
2. Exhaustive Explorer (탐험형)
페이지 전체를 훑어보는 성향

사용자 심리: "이 회사가 실제로 존재하는구나"
2. 🧭 보조 네비게이션 (Secondary Navigation)
메인 메뉴에 넣기 애매한 것들의 집합소
회사 소개, 채용 정보, 이용약관, 개인정보처리방침, 자주 묻는 질문, 고객센터, 파트너십, 제휴 문의
사용자 심리: "여기서는 뭐든 찾을 수 있겠네"
3. 🎯 전환 기회 (Conversion Opportunity)
마지막 설득의 기회
뉴스레터 구독, 앱 다운로드 유도, 소셜 미디어 팔로우, 프로모션 안내
사용자 심리: "더 볼 게 있나?"
