데이터 분석 도구와 활용 방법

사전 질문(15분)
서비스에서 발생하는 다양한 행동(예: 클릭, 방문, 구매)은 어떻게 데이터로 기록되고 분석될 수 있을까요?

특정 지표나 사용자 행동을 측정하려면, 기술적으로 어떤 과정을 거쳐야 할까요?

사전 질문 토론
1. 서비스에서 발생하는 다양한 행동(예: 클릭, 방문, 구매)은 어떻게 데이터로 기록되고 분석될 수 있을까요?
사용자의 행동은 로그로 자동 기록되며, 클릭·방문·전환 등 주요 이벤트를 기준으로 데이터가 수집된다.
이벤트는 JS 코드, SDK, 서버 로그 등 다양한 방식으로 추적되며 저장된 데이터는 분석 서버로 전송된다.
이후 데이터는 시간·사용자·행동 유형별로 가공되어 대시보드 등을 통해 시각화된다.
이를 기반으로 사용자 흐름, 유입경로, 전환율 같은 핵심 지표를 확인할 수 있게 된다.
2. 특정 지표나 사용자 행동을 측정하려면, 기술적으로 어떤 과정을 거쳐야 할까요?
먼저 측정 목적을 정의하고, 어떤 지표를 수집할지 이벤트 항목을 설계해야 한다.
이후 추적 코드를 서비스에 삽입하고, 수집된 로그를 저장 및 정제해 분석이 가능하도록 준비한다.
수집된 데이터는 지표 계산 및 대시보드 구현 등 분석 환경에서 활용된다.
정확한 측정을 위해 QA와 데이터 정합성 검증 절차도 반드시 포함되어야 한다.

Case: 데이터 분석은 어떤 도구로 해야 하나
데이터 분석 도구를 어떤것을 학습하고 활용해야 할까
고객 정보를 기반으로 제품 데이터를 분석해 인사이트를 발견하려고 할 때
Back-end 개발자에게 요청해 DB에서 주요 데이터를 가져오는 방식으로
어드민에서 관리하는 데이터를 엑셀로 받아서 직접 스프레드시트에서 분석
제품 개선을 위해 고객 행동을 분석해 이를 기반으로 인사이트를 발견하려고 할 때
Mixpanel의 행동 데이터를 기반으로 퍼널 분석을 수행해 고객 문제를 발견
매출이 잘 나오지 않는 포인트를 GA4를 통해 발견하고 이를 개선하기 위한 정보 수집
새로 개발한 기능이 실제로 고객들이 잘 사용하고자 하는지 확인하려고 할 때
A/B 테스트 분석을 위해 Amplitude의 테스트 대시보드를 구성
실제 히트맵 분석을 확인해 의사결정을 하기 위해 Hotjar를 사용

데이터 분석 도구가 인사이트를 결정하지 않는다
현대에는 고객 행동을 파악하기 위한 수많은 데이터 분석 도구가 발달
이런 도구를 활용하는 방법을 배우는 것은 PM으로서 데이터에서 확실한 인사이트를 얻기 위해서
어떤 도구를 배우는지 보다는 그 도구를 이용해서 인사이트를 어떻게 얻을 수 있는가가 더 중요하다

어떤 근무환경에서도 근본적으로 고객의 진짜 ‘행동'과 ‘감정'을 이해할 수 있는 방식으로 데이터를 수집하고 분석할 줄 알아야 한다.

주요 데이터 분석에 사용되는 도구들
SQL: 관계형 데이터베이스에서 구조화된 데이터를 조회하고 분석하는 표준 언어
Spreadsheet: Google Sheets, Excel 등으로 데이터를 가볍게 정리, 시각화, 계산할 수 있는 범용 도구
Google Analytics 4: 웹/앱의 유입, 유저 행동, 전환 흐름 등을 추적하는 구글의 공식 웹 분석 도구
Mixpanel / Amplitude: 이벤트 기반의 사용자 행동 분석 플랫폼으로, 퍼널 분석, 코호트, 리텐션 분석 등에 특화
Hotjar / FullStory: 사용자의 실제 사용 흐름을 시각적으로 확인할 수 있는 UX 리서치 도구 (히트맵, 세션 리플레이 등)

관계형 데이터베이스
왜 관계형 데이터베이스인가?
제품/서비스는 사용자, 행동, 로그 등 수많은 데이터를 구조화하여 저장
PM이 데이터를 읽고 분석하려면 DB 구조와 쿼리 방식 이해 필요
관계형 DB는 가장 널리 쓰이는 데이터 저장 및 조회 방식
관계형 DB의 주요 개념
테이블(Tables): 특정 유형의 데이터를 저장하는 구조 (예: users, events)
행(Rows): 개별 데이터 레코드 (예: 한 사용자, 하나의 이벤트)
열(Columns): 데이터 속성 (예: user_id, timestamp, event_type)
관계(Relationships): 테이블 간의 연결 (예: 사용자와 그들의 구매 내역)


일반적인 관계형 DB 구조
사용자 테이블 (Users)
각 행: 개별 사용자
주요 열: 사용자ID, 이름, 이메일, 가입일, 설정
이벤트 테이블 (Events)
각 행: 개별 사용자 행동
주요 열: 이벤트ID, 사용자ID, 이벤트 유형, 타임스탬프
제품/콘텐츠 테이블 (Products/Content)
각 행: 개별 상품/콘텐츠
주요 열: 제품ID, 이름, 설명, 가격, 카테고리
트랜잭션 테이블 (Transactions)
각 행: 개별 구매/구독
주요 열: 주문ID, 사용자ID, 금액, 상태, 날짜

SQL의 기본 개념과 역할
SQL(Structured Query Language)은 관계형 데이터베이스에서 데이터를 추출하고 분석하는 표준 언어입니다:
데이터 질의: 필요한 정보를 정확히 추출
데이터 필터링: 특정 조건에 맞는 데이터만 선별
데이터 집계: 합계, 평균, 최대값 등 계산
데이터 결합: 여러 테이블의 정보를 연결

SQL 기본 구조 이해하기
sql
SELECT [어떤 데이터를 가져올지]
FROM [어디서 데이터를 가져올지]
WHERE [어떤 조건으로 필터링할지]
GROUP BY [어떻게 데이터를 그룹화할지]
HAVING [그룹화 후 어떤 조건으로 필터링할지]
ORDER BY [어떤 순서로 결과를 정렬할지]

SQL의 기본 구조는 일상 언어와 유사
"무엇을 보고 싶습니까?" (SELECT)
"어디서 찾아야 합니까?" (FROM)
"어떤 조건을 만족해야 합니까?" (WHERE)
"어떻게 그룹화해서 볼까요?" (GROUP BY)
"어떤 순서로 정렬할까요?" (ORDER BY)

데이터 질문을 SQL로 변환하기
SQL 사고 과정:
필요한 정보: 가입 날짜, 사용자 ID, 이벤트 유형, 이벤트 발생 시간
데이터 소스: users 테이블, events 테이블
필터 조건: 지난 달 가입자, 가입 후 7일 이내 이벤트, 핵심 기능 이벤트
계산 방식: (조건 충족 사용자 수) ÷ (전체 신규 사용자 수)

SQL 쿼리 예시
sql
-- 지난 7일간 일별 활성 사용자 수
SELECT 
    DATE(event_time) as day,
    COUNT(DISTINCT user_id) as daily_active_users
FROM events
WHERE event_time >= CURRENT_DATE - 7
GROUP BY DATE(event_time)
ORDER BY day;
"무엇을 보고 싶습니까?" (SELECT)
"어디서 찾아야 합니까?" (FROM)
"어떤 조건을 만족해야 합니까?" (WHERE)
"어떻게 그룹화해서 볼까요?" (GROUP BY)
"어떤 순서로 정렬할까요?" (ORDER BY)

SQL만의 고유한 가치
1. 원천 데이터 직접 접근
가공되지 않은 원본 데이터에 접근 가능
데이터 수집 시점부터의 모든 정보 활용 가능
2. 유연한 데이터 조합
여러 테이블의 데이터를 자유롭게 결합
복잡한 조건과 필터 적용 가능

SQL만의 고유한 가치
3. 대용량 데이터 처리
스프레드시트로 처리하기 어려운 대규모 데이터셋 분석
서버에서 처리되어 로컬 리소스 부담 없음
4. 반복 가능한 분석
한번 작성한 쿼리를 저장하고 재사용 가능
자동화된 보고서와 대시보드의 기반

SQL 실무 접근 전략
1. 기본 패턴 먼저 마스터하기
자주 쓰이는 핵심 구문에 집중 → 반복 패턴 중심 학습
필수 구문:
SELECT, FROM, WHERE, GROUP BY, ORDER BY, JOIN, LIMIT
학습 패턴 예:
특정 조건에 맞는 유저 조회
날짜별 유입 수 집계
특정 행동 기준 사용자 세그먼트 추출
JOIN을 활용한 다중 테이블 분석
유저별 평균 행동 수 구하기 등

SQL 실무 접근 전략
2. 실제 제품 질문에 적용하기
추상적인 연습용 DB가 아닌, 제품의 진짜 지표를 뽑는 연습이 효과적
예시 질문
오늘 신규 가입자는 몇 명인가?
첫 로그인 후 3일 내 재방문율은?
장바구니에 담고 구매하지 않은 유저는 누구인가?
특정 이벤트 수행 비율이 가장 높은 디바이스는?
난이도 조절 팁:
쉬운 질문부터 시작 → 점점 조인, 집계, 조건 복잡성 추가
매 질문마다 "무슨 데이터를, 왜?" 필요한지 적어보기

SQL 실무 접근 전략
3. 데이터 모델 이해에 집중하기
테이블 간 관계(Relation)
어떤 테이블이 어떤 키(Primary/Foreign Key)로 연결되어 있는가?
예: users → orders → payments
데이터 사전(Data Dictionary)
각 테이블과 필드의 의미를 문서화한 자료
→ 용어 오해 없이 정확한 쿼리 작성 가능
ERD(Entity Relationship Diagram)
시각적으로 관계 파악 가능 → 쿼리 설계 시간 단축

SQL 실무 접근 전략
4. 협업을 통한 학습
데이터 팀과 페어 프로그래밍
정기적으로 세션 요청 → 30분이면 충분
실전 쿼리 예시를 보며 어떻게 쿼리를 구성하는지 흐름을 익힘
기존 쿼리 분석 → 수정 → 확장
예) 기존 유저 리스트 쿼리에 “최근 로그인” 조건 추가
집계 기준을 주별에서 일별로 변경 등
“이 쿼리 왜 이렇게 짰어요?” 질문하기
→ 비즈니스 로직과 데이터 구조 이해의 지름길

스프레드시트: 일상적인 분석 도구

스프레드시트의 실무적 가치:
진입장벽 없이 즉시 데이터 탐색 가능
SQL로 추출한 데이터의 추가 분석과 시각화
빠른 가설 검증과 프로토타입 대시보드 제작
팀과의 실시간 협업 및 인사이트 공유

Microsoft Excel vs Google Spreadsheet
Microsoft Excel 강점:
방대한 함수와 고급 분석 기능
대용량 데이터 처리 성능 우수(~1백만 행)
파워 피벗/쿼리, VBA 매크로로 고급 모델링
오프라인 작업 가능
Microsoft Excel 약점:
실시간 협업 제한적
라이센스 비용 발생
기기 간 동기화 어려움
Google Spreadsheet 강점:
뛰어난 실시간 협업 기능
무료 사용 가능
웹 기반으로 어디서나 접근 가능
Google 서비스와 원활한 통합
Google Spreadsheet 약점:
대용량 데이터 처리 시 성능 제한
고급 분석 기능 부족
인터넷 연결 필수

업무별 적합성 및 선택 가이드
Excel을 선택할 때:
복잡한 분석과 대용량 데이터 처리 필요 시
고급 시각화와 모델링 작업 시
오프라인 작업 환경에서
Google Sheets를 선택할 때:
실시간 팀 협업이 중요할 때
다양한 기기에서 접근이 필요할 때
간편한 공유와 피드백이 우선일 때
기본적인 분석으로 충분한 경우

핵심 기능과 활용법
1. 조건부 집계 함수
SUMIFS/COUNTIFS: 여러 조건을 충족하는 값의 합계/개수
AVERAGEIFS: 조건에 맞는 값들의 평균
MAXIFS/MINIFS: 조건부 최대/최소값
활용 예시: 모바일 신규 사용자의 전환율 계산
=COUNTIFS(user_type, "new", device, "mobile", conversion, "yes")/COUNTIFS(user_type, "new", device, "mobile")


핵심 기능과 활용법
2. 데이터 조회 및 매칭
VLOOKUP/HLOOKUP: 수직/수평 방향 데이터 조회
INDEX-MATCH: 더 유연한 데이터 조회
QUERY (Google Sheets): SQL과 유사한 문법으로 데이터 쿼리
활용 예시: 유료 채널별 총 매출 집계
=QUERY(A1:F1000, "SELECT A, SUM(F) WHERE B = 'paid' GROUP BY A LABEL SUM(F) 'Total Revenue'")


핵심 기능과 활용법
3. 피벗 테이블
다차원 데이터 집계 및 교차 분석
다양한 기준으로 데이터 분해 및 요약
인터랙티브한 데이터 탐색
피벗 테이블 활용 전략:
행/열 필드 계층화로 드릴다운 분석 (년→월→일)
값 표시 형식 최적화 (원본 값 대신 비율로 표시)
여러 피벗 테이블 연계로 종합 대시보드 구성


피벗 테이블 활용 사례
1. 코호트 분석
행: 가입 월 (코호트)
열: 경과 월 (1,2,3개월 등)
값: 활성 사용자 수 또는 유지율
2. 세그먼트 분석
행: 사용자 세그먼트 (지역, 플랜 등)
열: 핵심 지표 (활성도, 전환율 등)
값: 평균, 중앙값, 합계 등

피벗 테이블 활용 사례
3. 기능 채택 분석

데이터 추적 설계

1. 이벤트(Events)
사용자가 수행한 개별 행동을 나타냄
명확한 목적과 의미를 가져야 함
일관된 네이밍 규칙 필요 (예: 'Button_Click' 대신 'Subscription_Started')

분석 용이성을 고려한 세분화 수준

데이터 추적 설계
2. 속성(Properties/Parameters)
이벤트의 맥락과 세부 정보를 제공

공통 속성: 모든/다수 이벤트에 적용되는 정보 (예: 'page_name', 'screen_size')
사용자 속성: 사용자 수준의 정보 (예: 'subscription_plan', 'acquisition_source')

데이터 추적 설계 전략
2. 이벤트 및 속성 체계화
사용자 여정 매핑: 주요 단계와 상호작용 지점 식별
이벤트 계층 구조: 관련 이벤트 그룹화
속성 스키마: 일관된 속성 정의와 형식
이벤트 분류 체계 예시:
핵심 전환 이벤트: 'Signup_Completed', 'Subscription_Started'
참여 이벤트: 'Content_Viewed', 'Feature_Used'
수익 이벤트: 'Purchase_Completed', 'Renewal_Processed'
오류/마찰 이벤트: 'Error_Encountered', 'Process_Abandoned'

데이터 추적 설계 전략
3. 문서화 및 구현 가이드
표준화된 템플릿 활용
각 이벤트/속성의 명확한 정의 및 목적
개발자를 위한 기술적 구현 지침
예상 사용 사례와 분석 예시

데이터 추적 설계 전략
효과적 문서화 요소:
이벤트명: Feature_Used
설명: 사용자가 앱의 특정 기능을 사용했을 때 발생
발생 시점: 기능 사용 완료 시 (성공적인 결과 생성 후)
필수 속성:
  feature_name: 사용된 기능 이름 [string]
  feature_category: 기능의 카테고리 [string]
  interaction_method: 사용 방법 (버튼/단축키/음성) [string]
선택 속성:
  time_spent: 기능 사용에 소요된 시간(초) [number]
  content_id: 관련 콘텐츠 ID (해당시) [string]
분석 사용 사례:
  가장 인기 있는 기능 식별
  사용자 세그먼트별 기능 선호도 비교
  기능 사용과 유지율 간의 상관관계 분석

데이터 추적 설계 전략
4. 우선순위 설정 및 단계적 구현
핵심 사용자 여정 중심의 필수 이벤트 우선화
분석 가치와 구현 복잡성 균형 고려
반복적 접근: 기본 추적 구현 후 점진적 확장
우선순위 매트릭스:
1순위: 핵심 전환/수익 이벤트 (즉시 구현)
2순위: 주요 참여 및 사용자 경험 이벤트 (단기 구현)
3순위: 세부적 행동 및 보조 이벤트 (중기 구현)
4순위: 특수 분석용 고급 이벤트 (장기 구현)

분석 성숙도 모델
1. 기초 단계: 측정 시작
특징: 기본적인 사용자 행동 추적, 단순 지표 모니터링
도구: Google Analytics, 스프레드시트
조직 구조: PM이 직접 분석 담당, 전담 인력 없음
의사결정: 주로 직관 기반, 간단한 데이터 참조
발전 포인트:
핵심 이벤트와 전환 정의 및 추적
기본 대시보드 구축 및 정기 검토
데이터 기반 문화의 초기 토대 마련

분석 성숙도 모델
2. 발전 단계: 통찰 도출
특징: 다차원 분석, 세그먼트 비교, 코호트 분석
도구: 전문 분석 도구(Mixpanel, Amplitude), 기본 SQL
조직 구조: 분석 전담자 확보, PM-분석가 협업
의사결정: 데이터 인사이트 기반 가설 검증
발전 포인트:
표준화된 분석 프레임워크 도입
A/B 테스트 문화 정착
심층 분석을 위한 데이터 접근성 향상
정성적+정량적 데이터 통합 분석

분석 성숙도 모델
3. 성숙 단계: 예측과 최적화
특징: 행동 예측, 자동화된 최적화, 고급 분석
도구: 데이터 웨어하우스, 고급 BI 도구, ML 도구
조직 구조: 데이터 팀 구축, 전문 역할 분화
의사결정: 데이터 중심의 체계적 의사결정 프로세스
발전 포인트:
예측 모델을 통한 선제적 의사결정
개인화 및 실시간 최적화 구현
복잡한 인과관계 분석 및 검증
분석 결과의 자동화된 제품 피드백 루프

주요 분석 도구별 한계점
SQL
한계점: 복잡한 비즈니스 로직 표현의 어려움, 시각화 기능 부재, 비기술적 팀원과 공유 제한, 데이터 모델 의존성
보완 전략: 모듈화된 쿼리 작성, BI 도구와 연계, 주석 및 문서화 강화, 테이블 구조 이해에 투자
스프레드시트
한계점: 대용량 데이터 처리 한계, 복잡한 분석 제약, 실시간성 부족
보완 전략: 샘플 데이터 활용, 전문 도구와 병행, 자동화 파이프라인 구축
GA4
한계점: 샘플링 이슈(높은 트래픽), 익명 사용자 추적 제한, 개인 수준 분석 어려움
보완 전략: 데이터 스트림 세분화, 사용자 ID 활용 강화, BigQuery 연동 활용

주요 분석 도구별 한계점
Mixpanel/Amplitude
한계점: 유입 채널 분석 제한, 높은 구현 복잡성, 이벤트 설계 의존성
보완 전략: GA와 채널 데이터 연계, 점진적 이벤트 확장, 정기적 이벤트 감사
Hotjar/FullStory
한계점: 정성적 데이터 편향, 샘플 대표성 문제, 개인정보 처리 이슈
보완 전략: 충분한 샘플 확보, 정량 데이터와 교차 검증, 익명화 설정 강화

데이터 불확실성 대응 전략
다중 소스 검증(Triangulation)
동일한 지표를 여러 도구나 데이터 소스로 측정해 교차 검증
정량 데이터와 정성 피드백을 함께 분석해 해석의 깊이 확보
내부 데이터와 외부 벤치마크를 비교하여 인사이트 보완
데이터의 불확실성과 한계 인정
데이터의 한계와 불완전성을 명시적으로 설명
신뢰 구간, 오차 범위 등 불확실성의 크기까지 공유
"우리가 아는 것"과 "아직 모르는 것"을 구분해 커뮤니케이션
점진적 개선 접근법
완벽한 데이터를 기다리기보다 지금 가능한 데이터로 행동
가설 → 검증 → 조정을 반복하며 데이터 기반 결정을 고도화
수치 해석만이 아니라, 데이터 품질 자체의 개선도 함께 추진

개인 과제
자신의 프로젝트에서 검증하고 싶은 가설을 설정하고, 그 가설을 검증하기 위해 측정해야 할 핵심 지표를 정의해봅니다.(만약 가설이 없다면, 해당 제품의 핵심 가치를 가장 잘 보여주는 사용자 행동을 하나 선정해 지표로 설정해도 됩니다.)
해당 지표를 측정하기 위해 필요한 데이터는 무엇인지 구체적으로 정리해봅니다.(예: 어떤 이벤트가 필요하고, 어떤 속성이 함께 기록되어야 하는지)
그 데이터를 효과적으로 수집하기 위해 어떤 분석 도구를 사용할지 선택하고, 그 이유를 함께 설명해봅니다.
