# 제품 개발 라이프사이클

## 🚀 Quick Start
제품 개발 라이프사이클은 아이디어에서 출시, 성장, 그리고 때로는 단종까지 제품이 거치는 모든 단계를 포괄하는 체계적 프로세스입니다. PM은 이 전체 과정의 지휘자 역할을 담당합니다.

### 핵심 개념
- **제품은 지속적으로 진화**: 어떤 제품도 한 번에 완성되지 않음
- **PM은 전체 과정의 조율자**: 다양한 팀과 자원을 조화롭게 이끌어 제품 비전 실현
- **데이터 기반 의사결정**: 각 단계에서 측정과 학습을 통한 개선

## 📋 제품 관리자의 역할 진화

### 과거의 제품 관리자
- **초기 역할**: 마케팅과 제품 기획 사이의 중간자
- **초점**: 요구사항 문서화와 프로젝트 관리
- **의사결정**: 위계적, 경영진 주도
- **스킬셋**: 프로젝트 관리, 문서 작성, 기획

### 현대의 제품 관리자
- **현재 역할**: 제품 비전과 전략 주도자
- **초점**: 고객 문제 해결과 비즈니스 가치 창출
- **의사결정**: 데이터 기반, 협업적 접근
- **스킬셋**: 분석, 실험, 커뮤니케이션, 리더십, 기술 이해

> "전체 개발 프로세스에서 이런 문제들에 대해 책임감을 갖고 문제를 해결하는 역할"

## 🎯 기업 규모별 PM 역할 차이

### 스타트업
- **역할**: 다재다능한 제너럴리스트
- **특징**: 여러 역할 동시 수행
- **핵심**: "많은 일을 동시에 하며 빠르게 움직이는 사람"

### 중견기업
- **역할**: 특정 제품/기능에 집중
- **특징**: 전문성 개발
- **핵심**: "깊이와 넓이의 균형을 찾는 사람"

### 대기업
- **역할**: 전문화된 영역 담당
- **특징**: 복잡한 이해관계자 관리
- **핵심**: "영향력을 통해 일을 성사시키는 사람"

## 🔄 제품 개발 생명주기 5단계

### 1️⃣ 올바른 기회 찾기 및 계획하기
**목적**: 문제 정의와 기회 평가

#### 주요 활동
- 회사 전략적 맥락 이해
- 타겟 고객과 시장 분석
- 기회 가설 수립 및 검증
- MVP 정의와 범위 설정
- 성공 지표 설정 (SMART, AARRR)

#### PM의 역할
- 시장과 고객 니즈 파악
- 비즈니스 목표와 연계
- 우선순위 결정
- PRD 작성

### 2️⃣ 솔루션 설계하기
**목적**: 사용자 중심 솔루션 개발

#### 주요 활동
- 사용자 리서치 및 페르소나 개발
- 사용자 여정 매핑
- 정보 아키텍처 설계
- 프로토타이핑 (Lo-Fi → Mid-Fi → Hi-Fi)
- 사용성 테스트

#### PM의 역할
- 디자인팀과 사용자 리서치 협업
- 엔지니어링팀과 기술적 실현 가능성 확인
- 비즈니스 목표와 사용자 니즈 균형 조율
- 디자인 결정을 위한 데이터 제공

### 3️⃣ 솔루션 구축하기
**목적**: 개발 과정 관리

#### 주요 활동
- 개발 방법론 적용 (애자일, 스크럼, 칸반)
- 백로그 관리와 우선순위 설정
- 스프린트 계획과 실행
- 품질 관리와 테스트
- 기술적 부채 관리

#### PM의 역할
- 제품 백로그 관리
- 스토리 구체화
- 이해관계자와 개발팀 연결
- 진행 상황 추적과 장애물 제거

### 4️⃣ 솔루션 공유하기
**목적**: 출시 및 마케팅

#### 주요 활동
- 출시 전략 수립 (빅뱅/단계적/소프트 런칭)
- 메시징 전략 개발
- 출시 채널 선택과 실행
- 내부 팀 준비와 교육
- 출시 체크리스트 관리

#### PM의 역할
- 가치 제안 명확화
- 출시 계획 수립
- 마케팅팀과 협업
- 리스크 관리

### 5️⃣ 솔루션 평가하기
**목적**: 성과 측정 및 다음 단계 계획

#### 주요 활동
- 성과 지표 측정 (사용자/비즈니스/품질/만족도)
- 데이터 수집과 분석
- 팀 회고 진행
- 다음 단계 결정 (반복/확장/피벗/종료)

#### PM의 역할
- 데이터 분석과 인사이트 도출
- 회고 진행과 학습 문화 구축
- 전략적 방향 결정
- 로드맵 업데이트

## 💡 제품 삼각형 (Product Triangle)

제품 성공을 위한 세 가지 핵심 요소의 균형:

### 🏢 비즈니스 (Business)
- 회사 목표
- 수익 모델
- 경쟁 우위

### 💻 기술 (Technology)
- 기술적 가능성
- 인프라
- 엔지니어링 역량

### 👤 사용자 경험 (User Experience)
- 고객 니즈
- 사용성
- 감성적 연결

> PM은 이 세 요소 간의 균형을 맞추는 조정자 역할을 수행합니다.

## 🔀 개발 방법론의 진화

### 워터폴 (Waterfall)
- **특징**: 선형적, 단계별 접근
- **프로세스**: 요구사항 → 설계 → 구현 → 검증 → 유지보수
- **장점**: 명확한 계획, 예측 가능성
- **단점**: 유연성 부족, 늦은 피드백

### 린 (Lean)
- **특징**: 빠른 반복 주기, 학습 중심
- **프로세스**: Build → Measure → Learn
- **장점**: 빠른 시장 검증, 낮은 리스크
- **단점**: 장기 계획 부족 가능성

### 하이브리드
- **특징**: 워터폴과 린의 장점 결합
- **접근법**: "큰 그림은 계획하되, 실행은 반복적으로"
- **적용**: 
  - 장기적 제품 비전과 로드맵 수립 (워터폴)
  - 짧은 스프린트로 구현 및 검증 (린)

## 📊 성공 지표 프레임워크

### SMART 원칙
- **S**pecific (구체적)
- **M**easurable (측정 가능)
- **A**chievable (달성 가능)
- **R**elevant (관련성)
- **T**ime-bound (시간 제한)

### AARRR (해적 지표)
- **A**cquisition (획득): 사용자가 제품을 발견하는 방법
- **A**ctivation (활성화): 첫 긍정적 경험
- **R**etention (유지): 반복 사용
- **R**eferral (추천): 다른 사용자에게 추천
- **R**evenue (수익): 수익화

## 🚧 PM 역할에 대한 오해와 진실

### ❌ 오해
- "PM은 CEO의 지시를 개발팀에 전달하는 메신저다"
- "PM은 요구사항 문서만 작성하면 된다"
- "PM은 모든 결정의 최종 권한자다"
- "좋은 아이디어만 있으면 좋은 제품이 만들어진다"

### ✅ 진실
- PM은 다양한 인사이트를 종합하여 제품 방향성을 주도한다
- PM은 지속적으로 이해관계자와 소통하며 제품을 발전시킨다
- PM은 협업과 합의를 통해 결정을 이끌어내는 촉진자다
- 실행력, 검증, 반복개선이 아이디어만큼 중요하다

## 🎭 PM vs 유사 역할 구분

### Product Manager vs Project Manager
| 구분 | Product Manager | Project Manager |
|------|----------------|-----------------|
| 초점 | "무엇을" 만들지 | "어떻게" 만들지 |
| 책임 | 제품 성공, 고객 가치 | 프로젝트 실행, 일정 |
| 시간 | 중장기적 생명주기 | 정해진 프로젝트 기간 |
| 질문 | "고객에게 가장 가치 있는 것은?" | "어떻게 효율적으로 목표 달성?" |

### Product Manager vs Product Owner
| 구분 | Product Manager | Product Owner |
|------|----------------|---------------|
| 맥락 | 방법론 중립적 | 스크럼/애자일 특화 |
| 범위 | 전략, 로드맵, 비즈니스 | 백로그, 스프린트 |
| 초점 | 외부 (시장, 고객) | 내부 (개발팀) |
| 위치 | 전략적 | 전술적, 실행 중심 |

## 💼 실무 적용 가이드

### 회사 맥락 파악하기
1. **비전과 미션**: 왜 우리가 존재하는가?
2. **타겟 고객**: 누구를 위한 제품인가?
3. **핵심 역량**: 우리가 잘하는 것은?
4. **경쟁 환경**: 시장에서 우리의 위치는?
5. **비즈니스 모델**: 어떻게 수익을 창출하는가?

### 기회 가설 작성 템플릿
```
"우리는 [이 문제]가 [이 고객]에게 중요하며, 
[이 해결책]이 [이 비즈니스 가치]를 제공할 것이라고 믿는다"
```

### MVP 정의 프레임워크
1. **핵심 가치 명확화**: 제품이 해결하는 주요 문제 하나
2. **핵심 사용자 여정 매핑**: 가치 전달을 위한 최소 단계
3. **MoSCoW 방법론 적용**:
   - Must have: 필수 기능
   - Should have: 중요 기능
   - Could have: 있으면 좋은 기능
   - Won't have: 제외 기능

## 📚 사례 연구: 식료품 배달 앱

### 1단계: 기회 찾기
- **시장 조사**: 바쁜 전문직 타겟
- **기회 가설**: "신선한 식품 접근이 어려운 바쁜 전문직에게 빠른 배달 필요"
- **MVP**: 기본 상품 검색, 주문, 배달 추적

### 2단계: 설계
- **사용자 여정**: 발견 → 검색 → 주문 → 배달 → 재주문
- **프로토타입**: 와이어프레임 → 사용성 테스트

### 3단계: 구축
- **방법론**: 스크럼, 2주 스프린트
- **우선순위**: 핵심 기능 우선

### 4단계: 공유
- **출시**: 제한된 지역 소프트 런칭
- **메시지**: "바쁜 당신을 위한 30분 식료품 배달"

### 5단계: 평가
- **지표**: 주문량, 재주문율, 평균 주문 금액
- **다음 단계**: 배달 시간 최적화, 지역 확장

> "전 과정을 전부 직접 수행하지는 않지만, 전 과정 전부를 이해하고 책임감 있게 관리해야 한다"

제품 개발은 단순한 기능 구현이 아닌, 고객 가치 창출을 위한 지속적인 여정입니다. PM은 이 여정의 네비게이터로서, 팀과 함께 불확실성을 헤쳐나가며 성공적인 제품을 만들어갑니다.
